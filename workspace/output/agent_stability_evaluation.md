## 3. 산정 가정 (Assumptions)

- 동시접속률: DAU의 8% (재고관리 CRUD 업무 특성상 표준 가정)
- CAP 런타임(Node.js 기준) 1 동시접속당 필요 자원: 약 0.05 vCPU, 40MB 메모리
- HANA Cloud 메모리 요구량: 전체 사용자 수 × 8MB (마스터/트랜잭션 데이터 기준)
- CF 런타임 표준 인스턴스 티어: 1vCPU/2GB, 2vCPU/4GB
- Agent1: 버퍼 0%, 단일 인스턴스 우선(용량 초과 시에만 최소 스케일아웃)
- Agent2: 버퍼 50%, 최소 2 인스턴스(이중화) 이상 구성
- 실제 피크 부하는 평균 동시접속의 2~3배까지 튈 수 있음(가정) → 이 변동성이 여유율 없는 구성의 안정성을 크게 낮춤

## 6. 종합 비교

| 시나리오 | Agent1 (최소비용) | Agent2 (성능우선) | 격차 |
|---|---|---|---|
| S1 (DAU 120) | 52 | 88 | -36 |
| S2 (DAU 240) | 38 | 87 | -49 |
| S3 (DAU 360) | 24 | 83 | -59 |

**핵심 인사이트**
- Agent1은 규모(DAU)가 커질수록 안정성 점수가 급격히 하락(52→38→24). 여유율 없는 단일 인스턴스 구성이 부하 증가에 취약하기 때문.
- Agent2는 사용자 증가에도 비교적 안정적 점수(88→87→83)를 유지. 버퍼와 이중화 구조가 부하 변동을 흡수.
- 이중화 부재(Agent1)는 모든 시나리오에서 "장애/다운타임 리스크" 항목 점수를 가장 크게 낮추는 요인으로 작용.
- 사용자 규모가 커질수록 두 Agent 간 안정성 격차가 커짐 → 대규모 운영 환경에서는 비용 최소화 전략의 리스크가 기하급수적으로 커질 수 있음을 시사.

- 실제 운영 텔레메트리(APM 지표, 실측 응답시간 등)가 없어 모든 점수는 표준 가정 기반의 상대적 비교치임.
- 동시접속률(8%), 인당 자원 소비량, 버퍼 비율(0%/50%) 등은 업계 표준적 가정으로, 실제 시스템 프로파일링 결과와 다를 수 있음.
- 본 평가는 CAP Application의 "운영 안정성"에 한정되며, 기능 적합성/보안/데이터 정합성 등은 평가 범위 외.
